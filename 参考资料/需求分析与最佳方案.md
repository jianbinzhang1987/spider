# 电子元器件多渠道比价工具 — 需求分析与最佳方案

## 一、核心需求总结

根据需求文档，核心目标是：
1. 读取 Excel 采购清单（型号、品牌、采购数量）
2. 在 **18+ 个电子元器件网站**（国际3个 + 国内15个）抓取价格、库存、货期
3. 根据采购数量匹配阶梯价格
4. 美元汇率转换
5. 汇总比较，标记最低价
6. 输出 Excel 报告

## 二、当前实现存在的问题

### 2.1 严重问题

| 问题 | 影响 |
|------|------|
| **全部依赖 Playwright 浏览器自动化** | 速度极慢、资源消耗大、极易被反爬检测 |
| **反爬对抗能力弱** | 从 CSV 结果看，Digi-Key/Mouser/立创 大量返回"未找到型号"或"未显示价格" |
| **串行执行** | 5个网站×N个型号，全部串行，50个元件需要 30-60 分钟 |
| **密码硬编码在代码中** | 安全隐患，已提交到 Git |
| **仅实现 5/18 个网站** | 需求要求 18+ 个网站，覆盖率不足 |
| **需要人工介入验证码** | 无法自动化运行，不适合批量任务 |

### 2.2 架构问题

- 所有 Scraper 共享一个 Browser 实例 → 一个崩溃全部中断
- Streamlit 同步阻塞 UI → 用户体验差
- 缓存策略过于简单（只有全量 JSON 文件）
- 没有并发/并行机制
- 错误处理不完善，没有重试策略

### 2.3 数据质量问题（从实际运行结果 CSV 分析）

从 `2026-06-23T02-53_export.csv` 可以看出：
- **Digi-Key**: 几乎全部"未找到型号" → 搜索逻辑或反爬失败
- **Mouser**: 几乎全部"未找到型号" → API Key 未配置，浏览器路径被封
- **立创商城**: 能进入详情页但"未显示价格" → 价格提取选择器失效
- **华强电子网**: "无价格信息" → 搜索/提取逻辑有问题
- **云汉芯城 (ickey)**: 唯一能正常返回价格的，但价格异常低（0.001 元？）

## 三、最佳方案建议

### 3.1 整体架构：API优先 + 轻量HTTP + 浏览器兜底（三层策略）

```
┌─────────────────────────────────────────────────────┐
│                    调度层 (Orchestrator)               │
│  - 异步并发调度                                        │
│  - 每个网站独立任务，互不影响                            │
│  - 智能重试 + 熔断机制                                 │
└─────────────────┬───────────────────────────────────┘
                  │
    ┌─────────────┼─────────────────────┐
    ▼             ▼                     ▼
┌────────┐  ┌──────────┐  ┌──────────────────┐
│ 第一层  │  │  第二层   │  │     第三层        │
│ API调用 │  │ HTTP+解析 │  │ Playwright浏览器  │
│(最快最稳)│  │(快速可靠) │  │  (最后手段)       │
└────────┘  └──────────┘  └──────────────────┘
 Mouser API   立创/云汉等    需要JS渲染或
 Digi-Key API  可直接HTTP    登录态的网站
 (有官方API)   请求的网站
```

### 3.2 具体方案

#### 第一层：官方 API（推荐优先级最高）

| 网站 | API 可用性 | 说明 |
|------|-----------|------|
| **Mouser** | ✅ 有官方 Search API | 免费申请 API Key，无反爬问题 |
| **Digi-Key** | ✅ 有官方 Product API | 需要注册开发者账号，OAuth2 认证 |
| **element14** | ✅ 有官方 API | element14 Product Search API |
| **LCSC (立创)** | ⚠️ 有内部 API | 可抓取其前端调用的内部接口 |
| **云汉芯城** | ⚠️ 有内部 API | 搜索接口可直接 HTTP 调用 |

**优势**：速度快（<1秒/请求）、无反爬、数据结构化、稳定可靠

#### 第二层：HTTP 请求 + HTML 解析

对于没有公开 API 但页面可以通过简单 HTTP 请求获取的网站：
- 使用 `httpx` (async) 发送搜索请求
- 使用 `BeautifulSoup` / `parsel` 解析 HTML
- 携带必要的 Cookie/Session
- 比 Playwright 快 10-50 倍

适用网站：华强电子网、硬之城、猎芯网、华秋商城等

#### 第三层：Playwright 浏览器（仅作为兜底）

仅用于：
- 必须 JavaScript 渲染才能获取价格的网站
- 需要登录且有复杂验证的网站
- 前两层都失败时的降级方案

### 3.3 关键技术选型

| 方面 | 推荐方案 | 理由 |
|------|---------|------|
| **HTTP 客户端** | `httpx` (async) | 支持异步、HTTP/2、连接池 |
| **并发框架** | `asyncio` + `TaskGroup` | Python 原生异步，适合 I/O 密集 |
| **HTML 解析** | `parsel` 或 `BeautifulSoup` | 轻量、快速 |
| **浏览器自动化** | `playwright` (保留但降级使用) | 仅用于必须 JS 渲染的场景 |
| **前端 UI** | `Streamlit`（保留）或升级为 `FastAPI + 前端` | Streamlit 适合 MVP |
| **配置管理** | 环境变量 + `.env` 文件 | 不再硬编码密码 |
| **缓存** | SQLite 或 Redis | 比 JSON 文件更可靠、支持并发 |
| **汇率** | 保留当前方案 | 已经做了缓存，足够用 |

### 3.4 并发策略

```python
# 伪代码示意
async def scrape_all(parts_list, sites):
    results = []
    async with TaskGroup() as tg:
        for part in parts_list:
            for site in sites:
                tg.create_task(scrape_one(site, part))
    # 所有任务并行执行，按站点限流
```

- 每个网站设置独立的并发限制（Semaphore）
- 国际站点：2-3 并发
- 国内站点：3-5 并发
- 总体吞吐量提升 **10-20 倍**

## 四、关键改进点总结

| 改进 | 当前 | 建议 | 预期收益 |
|------|------|------|---------|
| 数据获取方式 | 全部 Playwright | API > HTTP > Playwright | 速度提升 10-50x，稳定性大幅提升 |
| 执行方式 | 同步串行 | 异步并发 | 速度提升 5-10x |
| 反爬处理 | 人工介入 | API 无需、HTTP 用 Session | 全自动运行 |
| 凭证管理 | 硬编码 | 环境变量 | 安全合规 |
| 缓存 | JSON 文件 | SQLite | 支持并发、查询方便 |
| 错误处理 | 简单 try/catch | 重试+熔断+降级 | 更高成功率 |
| 网站覆盖 | 5 个 | 分期扩展到 18 个 | 满足需求 |

## 五、实施建议

1. **先做 API 接入**：申请 Mouser API Key 和 Digi-Key API 开发者账号，这两个国际站的 API 文档完善、接入简单
2. **分析立创/云汉内部接口**：用浏览器 DevTools 抓取前端搜索请求，通常可以直接用 HTTP 复现
3. **搭建异步框架**：用 asyncio 重写调度层，让所有网站并行查询
4. **渐进式迁移**：新架构可以逐站替换，不需要一次全部重写

---

**总结**：当前方案最大的问题是「全部依赖 Playwright 浏览器自动化」，这在电商/元器件网站上注定不稳定。最佳方案是 **API 优先 + HTTP 请求为主 + 浏览器仅作兜底**，配合异步并发执行，可以在保证稳定性的同时将速度提升 1-2 个数量级。
